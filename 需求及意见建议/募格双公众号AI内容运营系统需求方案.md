# 募格双公众号 AI 内容运营系统需求方案

## 文档说明

**适用公众号**：

- 募格学术
- 募格科聘

**产品定位**：

> 面向募格旗下公众号矩阵的 AI 内容运营中台，基于历史文章、账号定位、读者反馈和运营数据，自动完成选题发现、资料整理、文章初稿、标题优化、公众号排版、草稿生成和数据复盘，关键节点由人工审核确认。

**建设原则**：

- AI 做收集、整理、生成、优化；
- 人做判断、审核、定调、发布；
- 不追求完全无人化发布；
- 优先建设“可控、可审、可复盘”的 AI 内容生产流程；
- 以历史文章和运营数据作为系统核心资产。

---

# 1. 项目背景

募格旗下已有多个面向科研群体的公众号，其中“募格学术”和“募格科聘”分别承担不同内容职能：

| 公众号 | 主要定位 | 核心读者 |
|---|---|---|
| 募格学术 | 学术资讯、科研生态、论文投稿、基金申报、研究生成长 | 研究生、博士、博士后、青年教师、科研人员 |
| 募格科聘 | 高校招聘、科研人才、博士就业、博士后机会、人才政策 | 博士、博士后、青年教师、科研求职者、高校招聘方 |

当前公众号内容运营中通常存在以下问题：

1. 选题依赖人工经验，容易临时化、重复化；
2. 查资料、整理背景、写初稿耗时较长；
3. 不同编辑写作风格不统一；
4. 历史文章没有充分资产化，复用效率低；
5. 公众号之间缺少联动机制；
6. 发布后数据复盘不够系统；
7. AI 可以提升效率，但内容风险需要人工控制。

因此，需要建设一个面向募格公众号矩阵的 AI 内容运营系统，将历史文章、内容流程、选题机制、写作规范和运营数据统一管理。

# 2. 产品目标

## 2.1 业务目标

### 目标一：提升内容生产效率

将传统内容流程：

```text
人工找选题 → 人工查资料 → 人工写稿 → 人工取标题 → 人工排版 → 人工复盘
```

升级为：

```text
AI 推荐选题 → 人工确认 → AI 整理资料 → AI 生成大纲和初稿 → 人工修改审核 → AI 排版成稿 → 人工发布 → AI 数据复盘
```

### 目标二：稳定内容供给

系统每日/每周自动生成候选选题池，为编辑团队持续提供：

- 近期热点选题；
- 长期 evergreen 选题；
- 历史文章翻新选题；
- 双账号联动选题；
- 节点型选题，如基金申报、毕业季、求职季、开学季等。

### 目标三：形成历史内容资产

将募格学术和募格科聘过往文章全部沉淀为知识库，用于：

- 查找历史内容；
- 避免重复选题；
- 学习账号风格；
- 分析爆款规律；
- 发现可翻新选题；
- 辅助新文章写作。

### 目标四：提升内容质量和一致性

通过账号风格模板、栏目模板、标题规则、审核规则，使 AI 生成内容更符合募格品牌风格。

### 目标五：降低内容风险

系统要在高风险内容上保留人工审核机制，尤其是涉及：

- 具体高校；
- 具体科研机构；
- 具体学者；
- 导师学生矛盾；
- 学术不端；
- 撤稿；
- 招聘岗位真实性；
- 薪资待遇；
- 编制信息；
- 政策解读；
- 医学健康；
- 法律争议。

# 3. 产品边界

## 3.1 系统要做什么

本系统主要完成：

1. 多公众号账号管理；
2. 历史文章知识库建设；
3. 选题发现与选题评分；
4. 双账号选题分发；
5. 资料包生成；
6. 文章大纲生成；
7. 文章初稿生成；
8. 标题、摘要、封面文案生成；
9. 内容审核和风险提示；
10. 公众号排版稿生成；
11. 草稿生成和发布前检查；
12. 数据复盘和策略建议。

## 3.2 系统不直接做什么

本系统初期不建议做：

1. 不建议完全无人审核自动发布；
2. 不建议自动发布涉及争议的人物、机构、政策类文章；
3. 不建议将 AI 生成内容直接作为最终稿；
4. 不建议对未经核实的招聘待遇、编制、薪资做确定性表述；
5. 不建议自动生成带有攻击性、指控性、定性结论的内容；
6. 不建议绕过公众号平台规则做自动化操作。

# 4. 用户角色与权限

## 4.1 运营负责人

负责整体内容方向、栏目规划、选题审核、发布节奏和数据复盘。

主要权限：

- 查看全部账号数据；
- 审核选题；
- 调整内容日历；
- 指派编辑任务；
- 查看文章风险报告；
- 进行最终发布确认；
- 查看运营复盘报告。

## 4.2 内容编辑

负责具体文章生产和修改。

主要权限：

- 查看选题池；
- 生成资料包；
- 生成文章大纲；
- 生成文章初稿；
- 修改文章；
- 优化标题和摘要；
- 生成排版稿；
- 提交审核。

## 4.3 审核人员

负责事实、引用、风险、合规审核。

主要权限：

- 查看待审核稿件；
- 查看引用来源；
- 查看风险标记；
- 标注问题；
- 退回修改；
- 审核通过。

## 4.4 管理员

负责系统配置和基础资料维护。

主要权限：

- 管理账号；
- 管理用户；
- 管理栏目；
- 管理模板；
- 管理敏感词；
- 管理历史文章库；
- 管理审核规则。

# 7. 多账号管理模块

## 7.1 功能定位

用于管理募格旗下不同公众号的定位、栏目、风格和运营规则。

系统至少支持两个账号：

- 募格学术
- 募格科聘

未来可扩展更多账号。

## 7.2 账号档案字段

每个公众号需要建立独立账号档案：

| 字段 | 说明 |
|---|---|
| 账号名称 | 募格学术 / 募格科聘 |
| 账号定位 | 一句话说明账号价值 |
| 核心读者 | 主要服务人群 |
| 栏目体系 | 固定栏目列表 |
| 内容禁区 | 不适合发布的内容 |
| 常用语气 | 严谨、理性、实用、共鸣、机会导向等 |
| 标题偏好 | 信息型、问题型、清单型、观点型等 |
| 发布频率 | 每日、工作日、每周等 |
| 审核等级 | 普通审核 / 严格审核 |
| 历史爆款特征 | 系统根据历史文章自动分析 |
| 风格示例文章 | 作为 AI 学习风格的参考样本 |

## 7.3 募格学术栏目建议

| 栏目 | 内容方向 |
|---|---|
| 科研热点 | 学术新闻、论文突破、科研动态 |

| 工具方法 | AI 工具、文献管理、统计工具、写作工具 |

# 8. 历史文章知识库模块

## 8.1 功能定位

将募格学术、募格科聘历史发布文章全部导入系统，形成可检索、可分析、可复用的内容资产库。

这是系统最重要的基础模块之一。

## 8.2 导入内容字段

每篇历史文章建议导入以下字段：

| 字段 | 说明 |
|---|---|
| 公众号名称 | 募格学术 / 募格科聘 |
| 文章标题 | 原始标题 |
| 发布时间 | 原始发布日期 |
| 正文内容 | 完整正文 |
| 文章摘要 | 原摘要或系统生成摘要 |
| 封面图 | 原封面图 |
| 作者/编辑 | 如果有 |
| 原始链接 | 文章链接 |
| 栏目标签 | 系统自动或人工标注 |
| 关键词 | 投稿、基金、博士后、招聘、导师等 |
| 内容类型 | 快讯、深度、经验、观点、招聘、政策等 |
| 阅读数据 | 阅读量、点赞、在看、收藏、转发等 |
| 留言数据 | 精选留言、用户反馈 |
| 风险标记 | 是否涉及投诉、争议、修订等 |
| 可复用程度 | 可直接复用 / 可翻新 / 可系列化 / 不建议复用 |

## 8.3 历史文章标签体系

建议给每篇文章打上多维标签：

| 标签类型 | 示例 |
|---|---|
| 账号标签 | 募格学术、募格科聘 |
| 栏目标签 | 论文投稿、博士就业、高校招聘 |
| 人群标签 | 硕士、博士、博士后、青椒、导师、HR |
| 场景标签 | 投稿、毕业、求职、基金、入职、晋升 |
| 风格标签 | 实用、评论、科普、深度、快讯、经验 |
| 风险标签 | 低风险、中风险、高风险 |
| 时效标签 | 长期有效、阶段有效、已过时 |
| 复用标签 | 可翻新、可拆分、可系列化、不可复用 |

## 8.4 历史文章分析能力

系统应能自动分析：

### 选题分析

- 过去高频选题；
- 长期表现稳定的选题；
- 曾经爆过但很久没写的选题；
- 最近重复过多的选题；
- 不同账号之间的内容重叠区域；
- 可改写为系列内容的选题。

### 标题分析

- 哪些标题结构阅读表现好；
- 哪些关键词点击表现好；
- 哪些标题容易引发评论；
- 哪些标题过于平淡；
- 哪些标题可能存在风险；
- 不同账号的标题风格差异。

### 风格分析

系统应总结两个账号的风格差异：

| 维度 | 募格学术 | 募格科聘 |
|---|---|---|
| 语言风格 | 学术、观察、实用、评论 | 实用、机会导向、职业规划 |
| 内容重点 | 论文、基金、科研生态、学术成长 | 岗位、招聘、求职、人才政策 |
| 情绪表达 | 适度共鸣，强调科研现实 | 强调机会、选择、行动建议 |
| 结构偏好 | 背景—分析—建议 | 机会/政策—条件—建议 |
| 读者利益点 | 提升科研能力、理解学术环境 | 找到机会、提高求职成功率 |

## 8.5 历史文章检索能力

编辑可以用自然语言搜索，例如：

- “找一下之前写过的国自然申请经验”
- “过去有哪些博士毕业压力相关爆款”
- “募格科聘有没有写过博士后出站选择”
- “最近一年是否写过高校非升即走”
- “找几篇适合翻新的投稿经验文章”
- “哪些招聘类文章阅读量最高”
- “哪些标题里带‘博士后’的文章表现最好”

系统返回结果应包括：

| 内容 | 说明 |
|---|---|
| 相关文章列表 | 标题、发布时间、账号 |
| 内容摘要 | 每篇文章讲了什么 |
| 数据表现 | 阅读、点赞、在看等 |
| 相似度说明 | 与当前查询的相关程度 |
| 可复用建议 | 是否适合重写、扩写、系列化 |
| 风险提示 | 是否过时、是否敏感、是否重复 |

# 9. 全网热点监控与选题拆解模块 (重构原选题发现模块)

## 9.1 模块定位与业务逻辑
本模块是内容生产的核心引擎。系统不再仅仅依赖历史文章生成选题[cite: 1]，而是通过“抓取全网热点 -> 结合历史知识库分析 -> AI 拆解多维选题”的数据管道（Data Pipeline）来驱动内容生成。

**核心工作流 (Workflow)**：
1. `Event_Trigger`: 定时任务或人工触发全网热点扫描。
2. `Hot_Topic_Extraction`: 聚合知乎、微博、各大高校论坛及学术资讯平台的热门榜单与高频词。
3. `History_Matching`: 将提取的热点事件传入「历史文章知识库模块」进行向量检索（Vector Search），拉取过往相关爆款数据。
4. `Topic_Decomposition`: 大模型结合热点信息与历史数据，将一个“大热点”拆解为适合“募格学术”和“募格科聘”不同栏目的多个“子选题”[cite: 1]。

## 9.2 数据结构定义 (Schema Definitions)

**HotTopicEvent (热点事件对象)**
- `event_id`: String (UUID)
- `event_title`: String (热点名称，如“某高校实行新型长聘制”)
- `heat_index`: Integer (全网热度指数)
- `source_urls`: List[String] (热点溯源链接)
- `keywords`: List[String] (核心标签)

**DecomposedTopic (拆解后的子选题对象)**
- `topic_id`: String (UUID)
- `parent_event_id`: String (关联的 HotTopicEvent ID)
- `target_account`: Enum("募格学术", "募格科聘", "双账号联动")
- `angle_description`: String (切入角度说明)
- `historical_reference`: List[String] (参考的募格历史爆款文章 ID)
- `expected_risk`: Enum("LOW", "MEDIUM", "HIGH")

## 9.3 选题拆解示例 (Codex Prompt Logic)

**输入 (Input):**
- 全网热点：近期某顶尖高校宣布大幅提高博士后待遇并改革考核机制。
- 历史数据匹配结果：去年发布过“博士后薪资大起底”（高阅读），前年发布过“非升即走下的博士后生存现状”（高评论）。

**输出 (Output - DecomposedTopics):**
1. **子选题 A (募格学术)**：从学术内卷与评价体系改革角度切入，探讨新型考核机制对科研产出的影响。
2. **子选题 B (募格科聘)**：梳理该高校最新博士后待遇细节，对比同量级高校，给出求职投递建议[cite: 1]。

# 10. 外网顶刊与撤稿前沿监控模块 (新增)

## 10.1 模块定位
负责每日自动化检索全球顶尖学术期刊及学术规范监督网站的最新动态，通过自动化翻译降低编辑团队的信息获取门槛，形成“待认领”的学术前沿素材库。

## 10.2 监控源与抓取策略 (Scraping Strategy)

| 目标源 (Source) | 抓取频率 (Cron) | 抓取目标 (Target Elements) |
|---|---|---|
| Nature (News & Articles) | 每日 08:00 | 最新发文、科技新闻板块 |
| Science (News & Articles)| 每日 08:00 | 最新发文、科技新闻板块 |
| Cell (Articles) | 每日 08:00 | 最新发文 |
| Retraction Watch | 每日 08:00, 14:00 | 最新撤稿通报、学术不端调查新闻 |

## 10.3 自动化翻译与清洗管道 (ETL Pipeline)

1. **Extract**: 爬虫服务 (`ScraperService`) 获取原文 Title（标题）、Abstract（摘要）、Authors（作者）、Publish Date（发布日期）和 DOI/URL。
2. **Transform (LLM Translation)**: 调用大模型翻译接口，限定 Prompt 输出精炼的学术中文。
   - 要求：保留核心专业术语，摘要翻译控制在 200 字以内，概括核心发现或撤稿原因。
3. **Load**: 存入系统的 `AcademicMonitorQueue`（待审队列）。

## 10.4 监控对象数据结构 (AcademicMonitorItem)
- `item_id`: String
- `source_platform`: Enum("Nature", "Science", "Cell", "RetractionWatch")
- `original_title`: String (英文原标题)
- `translated_title`: String (中文翻译标题)
- `original_abstract`: Text
- `translated_summary`: Text (AI 提炼的中文简要)
- `original_url`: String
- `status`: Enum("UNREAD" 未读, "CONVERTED" 已转选题, "DISCARDED" 忽略)

## 10.5 编辑人工审核流 (Human-in-the-loop)
提供专门的 **前沿资讯看板 (Dashboard)**：
- 编辑每日浏览 `translated_title` 和 `translated_summary`。
- 操作选项 1：**一键转为选题 (Convert to Topic)** -> 系统自动带着该条目的内容跳转至“资料整理模块”[cite: 1]，并打上“高风险/严格审核”标签（尤其是 Retraction Watch 内容）[cite: 1]。
- 操作选项 2：**忽略 (Discard)** -> 归档，不再提示。

# 11. 选题分发与双账号联动模块

## 11.1 功能定位

系统判断一个选题更适合哪个公众号，以及是否适合两个账号联动发布。

## 11.2 选题分发规则示例

| 选题 | 推荐账号 | 原因 |
|---|---|---|
| 国自然申请书常见问题 | 募格学术 | 更偏科研申报 |
| 博士如何准备高校面试 | 募格科聘 | 更偏求职就业 |
| 非升即走下的青年教师压力 | 募格学术 / 募格科聘 | 既有学术生态，也有职业选择 |
| 某高校招聘青年人才 | 募格科聘 | 明确招聘内容 |
| 撤稿事件带来的学术规范反思 | 募格学术 | 更偏学术规范 |
| 博士后出站后去高校还是企业 | 募格科聘 | 更偏职业发展 |

## 11.3 双账号联动示例

### 示例一：青年教师压力

**募格学术版本**：

标题方向：

> 青年教师为什么越来越难？科研、教学与考核压力下的现实困境

重点讲：

- 科研评价；
- 论文压力；
- 基金压力；
- 青年教师学术成长。

**募格科聘版本**：

标题方向：

> 博士进高校前，必须想清楚这几个现实问题

重点讲：

- 岗位选择；
- 考核要求；
- 编制与非编；
- 薪资待遇；
- 职业发展路径。

### 示例二：博士后选择

**募格学术版本**：

标题方向：

> 博士后阶段，到底是不是科研成长的关键跳板？

重点讲：

- 科研训练；
- 论文产出；
- 基金申请；
- 学术发展。

**募格科聘版本**：

标题方向：

> 博士后出站后，如何选择高校、科研院所和企业？

重点讲：

- 求职路径；
- 岗位匹配；
- 待遇比较；
- 出站规划。

# 12. 资料整理模块

## 12.1 功能定位

选题确认后，系统自动生成资料包，供编辑写稿和审核使用。

## 12.2 资料包内容

每个资料包包括：

| 内容 | 说明 |
|---|---|
| 选题背景 | 这个话题为什么重要 |
| 核心事实 | 时间、人物、机构、政策、论文、数据 |
| 关键观点 | 可展开的分析方向 |
| 参考资料 | 论文、政策、新闻、历史文章 |
| 历史相关文章 | 募格过去是否写过 |
| 读者痛点 | 读者最关心的问题 |
| 可用案例 | 可引用的案例 |
| 争议点 | 是否存在不同观点 |
| 风险提示 | 哪些表述需要谨慎 |
| 文章角度建议 | 从哪个角度写更适合该账号 |
| 可视化建议 | 是否适合表格、流程图、清单 |

## 12.3 资料可信度分级

| 等级 | 来源 | 使用规则 |
|---|---|---|
| A 级 | 官方文件、论文原文、期刊官网、高校官网、招聘官网 | 可作为核心事实 |
| B 级 | 主流媒体、行业媒体、专家公开发言 | 可作为辅助材料 |
| C 级 | 自媒体、论坛、社交媒体 | 只能作为舆论观察 |
| D 级 | 截图、匿名爆料、来源不明内容 | 原则上不得直接使用 |

## 12.4 资料包审核要求

以下类型选题的资料包必须人工确认后才能进入写作：

- 涉及具体高校；
- 涉及具体导师、学生、学者；
- 涉及撤稿、学术不端、举报；
- 涉及政策解读；
- 涉及招聘岗位、待遇、编制；
- 涉及未公开信息；
- 涉及争议性评价。

# 13. AI 写作模块

## 13.1 功能定位

根据选题和资料包，生成符合对应公众号风格的文章大纲、初稿和修改稿。

## 13.2 写作流程

```text
确认选题
  ↓
生成资料包
  ↓
选择文章类型
  ↓
生成大纲
  ↓
人工确认大纲
  ↓
生成初稿
  ↓
编辑修改
  ↓
AI 辅助润色
  ↓
进入审核
```

## 13.3 支持文章类型

| 类型 | 适合账号 | 说明 |
|---|---|---|
| 快讯型 | 两个账号 | 适合热点、招聘信息、政策变化 |
| 深度解读型 | 募格学术 | 适合科研热点、学术政策、论文事件 |
| 经验指南型 | 两个账号 | 适合投稿、基金、求职、面试、读博 |
| 清单型 | 两个账号 | 适合工具、避坑、流程、准备事项 |
| 观点评论型 | 募格学术 | 适合科研生态、评价体系、学术争议 |
| 招聘解读型 | 募格科聘 | 适合高校岗位、人才政策、博士后机会 |
| 系列文章型 | 两个账号 | 适合长期栏目化运营 |

## 13.4 募格学术写作要求

文章应做到：

- 有学术信息密度；
- 有背景解释；
- 有理性分析；
- 有科研人员视角；
- 不轻易下结论；
- 避免过度情绪化；
- 涉及争议时保持中性；
- 引用论文、政策、机构信息时明确来源。

适合结构：

```text
发生了什么
为什么重要
背后反映了什么问题
对科研人员有什么影响
我们应该如何看待
```

## 13.5 募格科聘写作要求

文章应做到：

- 强调实用性；
- 强调岗位、机会、路径；
- 语言更直接；
- 给读者行动建议；
- 对招聘信息核准来源；
- 对薪资待遇、编制、平台资源等谨慎表述。

适合结构：

```text
机会/问题是什么
适合哪些人
需要关注哪些条件
申请/准备时要注意什么
给求职者的建议
```

## 13.6 文章生成字段

每篇文章生成时应包括：

| 字段 | 说明 |
|---|---|
| 正文标题 | 主标题 |
| 备选标题 | 至少 5 个 |
| 公众号摘要 | 文章卡片摘要 |
| 文章导语 | 正文开头 |
| 正文内容 | 完整文章 |
| 小标题 | 自动分级 |
| 重点句 | 可加粗或做成金句 |
| 引用资料 | 与正文对应 |
| 风险提示 | 哪些段落需要人工确认 |
| 推荐配图 | 封面和正文配图建议 |
| 推荐标签 | 内容标签 |
| 推荐发布时间 | 根据账号节奏建议 |

# 14. 标题、摘要与封面文案模块

## 14.1 功能定位

为文章生成更适合公众号传播的标题、摘要、封面文案和社群转发语。

## 14.2 标题生成要求

每篇文章至少生成 10 个标题，分为：

| 类型 | 说明 |
|---|---|
| 稳健型标题 | 准确、克制、适合严肃内容 |
| 传播型标题 | 更有点击欲望，但不标题党 |
| 问题型标题 | 用问题引发读者兴趣 |
| 清单型标题 | “这几点”“这几个问题” |
| 情绪共鸣型标题 | 适合读博、就业压力类内容 |
| 观点型标题 | 适合学术观察和职业观察 |
| 信息型标题 | 适合招聘、政策、快讯内容 |

## 14.3 标题评分维度

| 维度 | 说明 |
|---|---|
| 准确性 | 是否符合正文内容 |
| 点击潜力 | 是否能吸引目标读者 |
| 账号匹配度 | 是否符合募格学术/募格科聘风格 |
| 风险程度 | 是否夸张、误导、容易引发争议 |
| 历史相似度 | 是否类似过去表现好的标题 |
| 新鲜感 | 是否避免老套表达 |

## 14.4 标题禁用规则

标题不得：

- 编造结论；
- 夸大科研结果；
- 使用未经证实的指控；
- 对具体人物或机构进行定性攻击；
- 使用过度煽动性表达；
- 把相关性说成因果关系；
- 把个案说成普遍现象；
- 对岗位待遇、编制和薪资做未经核实的承诺式表述。

## 14.5 摘要与转发语

系统应生成：

| 内容 | 用途 |
|---|---|
| 公众号摘要 | 显示在文章卡片 |
| 正文导语 | 文章开头 |
| 朋友圈转发语 | 编辑或作者转发使用 |
| 社群转发语 | 微信群、社群分发 |
| 海报短文案 | 配合封面图或海报 |

# 15. 内容审核与风控模块

## 15.1 功能定位

确保 AI 生成内容可以安全进入人工发布流程。

这是整个系统最重要的模块之一。

## 15.2 审核类型

| 审核类型 | 说明 |
|---|---|
| 事实审核 | 检查人物、机构、时间、数据、政策、论文是否准确 |
| 引用审核 | 检查是否有来源、来源是否可靠 |
| 重复审核 | 检查是否与历史文章过度相似 |
| 风格审核 | 是否符合账号定位 |
| 敏感审核 | 是否涉及争议、投诉、法律风险 |
| 招聘审核 | 岗位、待遇、编制、单位信息是否需要人工确认 |
| 学术规范审核 | 是否存在不当引用、过度解读、夸大研究结论 |
| 版权审核 | 是否存在图片、表格、原文大段复制风险 |

## 15.3 风险等级

| 等级 | 含义 | 处理方式 |
|---|---|---|
| 低风险 | 普通经验、工具、方法类内容 | 编辑审核即可 |
| 中风险 | 涉及政策、招聘、机构、待遇、学术评价 | 审核人员确认 |
| 高风险 | 涉及具体人物、举报、撤稿、争议、指控 | 必须负责人终审 |

## 15.4 高风险内容清单

以下内容应强制进入人工严格审核：

1. 学术不端；
2. 论文撤稿；
3. 导师学生矛盾；
4. 对具体高校或机构的负面评价；
5. 对具体学者的评价；
6. 招聘待遇、编制、薪资；
7. 人才政策解读；
8. 医学健康建议；
9. 法律纠纷；
10. 未公开消息；
11. 用户爆料；
12. 社交平台截图。

## 15.5 审核结果

审核人员可以对稿件执行：

| 操作 | 说明 |
|---|---|
| 通过 | 进入排版或发布前确认 |
| 修改后通过 | 编辑修改指定问题后通过 |
| 退回重写 | 文章问题较大，需要重写 |
| 延期发布 | 资料不充分或时机不合适 |
| 放弃发布 | 风险过高或选题价值不足 |

# 16. 公众号排版与草稿模块

## 16.1 功能定位

将审核通过的文章转换成适合微信公众号发布的排版稿，并生成草稿。

## 16.2 排版要求

系统应支持：

- 一级标题、二级标题样式；
- 重点句加粗；
- 引用块；
- 分割线；
- 表格；
- 图片占位；
- 文末参考资料；
- 文末引导关注；
- 文末招聘或社群入口；
- 公众号摘要；
- 封面图建议；
- 分享卡片标题。

## 16.3 不同账号排版差异

| 项目 | 募格学术 | 募格科聘 |
|---|---|---|
| 正文风格 | 稳重、清晰、学术感 | 清晰、实用、信息导向 |
| 重点突出 | 核心观点、研究结论、方法建议 | 岗位条件、申请建议、关键提醒 |
| 结尾引导 | 关注科研成长、留言讨论 | 关注岗位信息、投递建议、求职服务 |
| 表格使用 | 适合论文对比、方法步骤 | 适合岗位条件、时间节点、申请材料 |

## 16.4 草稿生成前检查

生成公众号草稿前必须检查：

- 标题是否为空；
- 摘要是否为空；
- 封面图是否设置；
- 正文是否有明显格式错误；
- 引用来源是否齐全；
- 高风险标记是否处理；
- 是否存在未确认事实；
- 是否通过审核。

# 18. 数据复盘模块

## 18.1 功能定位

文章发布后，系统收集或录入运营数据，生成复盘报告，并反哺选题和标题策略。

## 18.2 数据字段

每篇文章建议记录：

| 字段 | 说明 |
|---|---|
| 公众号 | 募格学术 / 募格科聘 |
| 标题 | 发布标题 |
| 发布时间 | 实际发布时间 |
| 栏目 | 所属栏目 |
| 阅读量 | 文章阅读数据 |
| 点赞量 | 点赞数据 |
| 在看量 | 在看数据 |
| 收藏量 | 收藏数据 |
| 转发量 | 转发数据 |
| 评论量 | 留言互动数据 |
| 新增关注 | 文章带来的关注增长 |
| 取消关注 | 发布后流失情况 |
| 打开率 | 如果可获得 |
| 分享率 | 如果可获得 |
| 完读率 | 如果可获得 |
| 转化数据 | 招聘、社群、课程、服务等转化 |

## 18.3 复盘分析内容

系统应自动生成：

- 本文表现等级；
- 标题表现分析；
- 选题表现分析；
- 发布时间表现分析；
- 留言情绪分析；
- 是否适合二次传播；
- 是否适合改写成系列文章；
- 是否适合另一个账号联动；
- 下一篇延展选题建议。

## 18.4 数据反哺机制

复盘结果应影响后续：

| 反哺对象 | 影响内容 |
|---|---|
| 选题推荐 | 提升表现好主题的权重 |
| 标题生成 | 学习高表现标题结构 |
| 账号风格 | 调整表达方式和文章长度 |
| 内容日历 | 优化发布频率和发布时间 |
| 双账号联动 | 发现适合转化到另一个账号的内容 |

# 19. 跨账号内容复用模块

## 19.1 功能定位

将一个账号的优秀内容转化为另一个账号适合发布的版本。

## 19.2 复用方式

| 复用方式 | 说明 |
|---|---|
| 原题改角度 | 同一选题，从不同账号定位重写 |
| 爆款翻新 | 将旧爆款结合新情况重新写 |
| 系列拆分 | 将长文拆成多篇系列文 |
| 清单化改写 | 将深度文改成实用清单 |
| 招聘化改写 | 将学术话题转为职业发展话题 |
| 学术化改写 | 将招聘话题转为科研生态观察 |

## 19.3 复用示例

### 从募格学术到募格科聘

原选题：

> 青年教师为什么越来越难？

募格科聘改写方向：

> 博士进高校前，必须了解青年教师的真实考核压力

### 从募格科聘到募格学术

原选题：

> 这些高校正在招聘博士后

募格学术改写方向：

> 博士后阶段如何影响科研职业路径？

# 20. 人工审核机制

## 20.1 必须人工确认的节点

| 节点 | 是否必须人工确认 |
|---|---|
| 选题入选 | 是 |

| 初稿修改 | 是 |
| 风险审核 | 是 |
| 发布前终审 | 是 |
| 实际发布 | 是 |

## 20.2 审核流转

```text
编辑提交
  ↓
系统自动检查
  ↓
低风险：编辑复核
中风险：审核人员复核
高风险：运营负责人终审
  ↓
通过后进入排版和草稿
```

## 20.3 审核记录

每次审核应保留：

- 审核人；
- 审核时间；
- 审核意见；
- 修改记录；
- 风险标记；
- 最终处理结果。

# 21. 典型使用场景

## 场景一：每日自动选题

每天上午，系统为两个账号各生成 5-10 个选题。

运营负责人看到：

| 选题 | 推荐账号 | 推荐理由 | 风险等级 |
|---|---|---|---|
| 青年基金申请书常见问题 | 募格学术 | 申报季需求强 | 低 |
| 博士后出站后如何找高校岗位 | 募格科聘 | 求职需求强 | 低 |
| 某高校人才政策变化 | 募格科聘 | 时效性强 | 中 |
| 近期撤稿事件反思 | 募格学术 | 讨论度高 | 高 |

负责人选择其中 2-3 个进入写作流程。

## 场景二：历史爆款翻新

编辑搜索：

> 去年表现好的投稿经验文章有哪些可以重写？

系统返回：

- 原文章列表；
- 阅读表现；
- 发布时间；
- 是否过时；
- 可翻新角度；
- 新标题建议。

## 场景三：双账号联动

系统发现“博士后是否值得做”适合两个账号：

- 募格学术：从科研成长角度写；
- 募格科聘：从就业选择角度写。

系统自动生成两篇不同角度的大纲。

## 场景四：招聘文章审核

募格科聘生成一篇高校招聘解读文章。

系统自动标记：

- 薪资待遇需要核实；
- 编制信息需要核实；
- 申请截止时间需要核实；
- 岗位来源必须为官网或官方公告；
- 不得使用“ guaranteed 录用”“待遇极高”等承诺式表达。

# 22. MVP 版本建议

建议分三期建设。

## 22.1 第一阶段：AI 编辑助理版

目标：先让编辑团队用起来，提高写作效率。

核心功能：

1. 历史文章导入；
2. 历史文章检索；
3. 单篇选题生成资料包；
4. 文章大纲生成；
5. 初稿生成；
6. 标题和摘要生成；
7. 基础风险提示；
8. Markdown/公众号排版稿输出。

暂不强求：

- 完整数据复盘；
- 自动草稿生成；
- 多账号复杂联动；
- 全自动抓取外部热点。

## 22.2 第二阶段：多账号运营中台版

目标：统一管理募格学术和募格科聘。

新增功能：

1. 多账号管理；
2. 每日选题池；
3. 选题评分；
4. 双账号选题分发；
5. 内容日历；
6. 更完整的审核流程；
7. 历史爆款分析；
8. 跨账号内容复用。

## 22.3 第三阶段：数据驱动运营版

目标：形成 AI 运营闭环。

新增功能：

1. 发布数据自动或半自动采集；
2. 标题表现分析；
3. 栏目表现分析；
4. 读者留言分析；
5. 选题策略自动调整；
6. 爆款预测；
7. 月度运营报告；
8. 自动推荐下一轮内容计划。

# 25. 内容风险原则

## 25.1 学术类内容原则

募格学术文章应遵守：

- 不虚构论文；
- 不虚构数据；
- 不夸大科研成果；
- 不将相关性说成因果关系；
- 不对具体学者做未经证实的评价；
- 不传播未经核实的举报和爆料；
- 不直接复制论文或媒体原文大段内容；
- 涉及争议时尽量使用中性表达。

## 25.2 招聘类内容原则

募格科聘文章应遵守：

- 岗位信息必须可追溯；
- 截止时间必须核实；
- 薪资待遇必须谨慎表述；
- 编制、安家费、科研启动费等必须注明来源；
- 不得承诺录用；
- 不得夸大岗位优势；
- 不得隐瞒重要限制条件；
- 不得使用未经授权的单位招聘信息做误导性包装。

## 25.3 标题原则

两个账号标题都应避免：

- 过度标题党；
- 恐吓式表达；
- 未证实指控；
- 绝对化结论；
- 对立情绪煽动；
- 误导读者理解文章内容。

# 28. 最终建议

本系统最合理的形态不是“全自动运营公众号”，而是：

> AI 公众号运营中台 + 历史内容知识库 + 多账号风格引擎 + 人工审核工作流。

建议优先从第一阶段 MVP 开始：

1. 先导入募格学术和募格科聘历史文章；
2. 建立两个账号的栏目和风格模板；
3. 实现历史文章检索、选题资料包、初稿生成、标题优化和排版输出；
4. 保留人工审核；
5. 等编辑团队稳定使用后，再扩展到自动选题池、内容日历、数据复盘和双账号联动。

这样既能快速看到效率提升，又能控制内容风险，并逐步把募格过去积累的文章真正变成可持续复用的内容资产。
